# SupportNest Outbound Events

SupportNest publishes tenant-scoped outbound events after committed domain mutations. Events are persisted in `outbound_events` first, then delivered by Active Job in local mode or by the dedicated outbox relay in production-style mode.

This document describes the current runtime contract. It does not introduce a new runtime dependency or change dispatch behavior.

## Runtime flow

- Domain services call `Events::Publisher.publish!` inside the same database transaction as the business mutation.
- `OutboundEvent` stores aggregate metadata, tenant ownership, correlation metadata, an idempotency key, and event-specific payload.
- Active Job mode enqueues `OutboundEventDispatchJob` after commit when `OUTBOX_DISPATCH_MODE=active_job`.
- Relay mode leaves events pending until `bin/outbox relay --loop` claims due rows with PostgreSQL `FOR UPDATE SKIP LOCKED`.
- Delivery is at least once; consumers must deduplicate by `id` or `idempotency_key`.
- Failed events move through retry/backoff and eventually to dead letter state.
- Replay creates a new pending outbound event with `replayed_from_outbound_event_id` lineage and a new replay idempotency key.

## Delivery payload

The webhook body is generated by `OutboundEvent#delivery_payload`.

Required fields:

- `id`: outbound event row identifier and stable delivery event id
- `event_type`: dotted event type
- `aggregate_type`: Rails model class name for the changed aggregate
- `aggregate_id`: aggregate primary key
- `organization_id`: tenant boundary for partitioning and authorization context
- `idempotency_key`: producer-generated key for consumer deduplication
- `correlation_id`: request or workflow correlation value
- `payload`: event-specific JSON object
- `created_at`: producer timestamp in ISO 8601 format

Optional fields:

- `replayed_from_outbound_event_id`: original failed event id when the delivery is an operator replay

## Supported event types

- `organization.bootstrapped`
- `membership.created`
- `membership.updated`
- `membership.token_revoked`
- `membership.token_rotated`
- `ticket.created`
- `ticket.updated`

## Webhook headers

- `Content-Type: application/json`
- `Idempotency-Key`: mirrors the event `idempotency_key`
- `X-SupportNest-Event-ID`: outbound event id
- `X-SupportNest-Event-Type`: event type
- `X-SupportNest-Signature-Timestamp`: Unix timestamp used in the signature base string
- `X-SupportNest-Signature`: HMAC SHA-256 over `timestamp.body`

## Consumer responsibilities

- Treat delivery as at least once.
- Deduplicate by `Idempotency-Key` or `X-SupportNest-Event-ID`.
- Verify the HMAC signature before accepting the payload.
- Persist the last accepted event id or idempotency key before running side effects.
- Use `organization_id` as tenant context, not as proof of authorization.
- Reject unknown event types unless an explicit compatibility policy says otherwise.

## Compatibility policy

- Additive payload fields are backward-compatible.
- Removing or renaming existing fields requires a new schema version and consumer rollout plan.
- New event types require documentation, test coverage, and runbook updates.
- Replay must remain explicit and operator-driven through `bin/outbox replay`.
- External brokers such as SQS, RabbitMQ, Kafka, or Sidekiq may replace the delivery transport later, but the persisted outbox state machine remains the source of truth.
